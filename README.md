# ESP32 WiFi Security Suite V2.1 (Aggressive Core)

A sophisticated, hardware-integrated Wi-Fi security assessment tool built on the ESP32 platform. This suite features dual-band/multi-channel packet injection capabilities, an RFID-based physical access control lock (MFRC522), a captive portal for credential harvesting in authorized "Evil Twin" scenarios, and a responsive web dashboard for administration.

> [!WARNING]
> This tool is designed strictly for educational purposes, authorized security auditing, and wireless penetration testing of networks you own or have explicit permission to test. Unauthorized use on public or third-party networks is illegal.

---

## 🚀 Key Features

*   **Non-blocking Attack Engine (V7):** Safely multiplexes packet transmission loop with the web server and system tasks. Built to be FreeRTOS-watchdog friendly to prevent system crashes during heavy traffic floods.
*   **Five Specialized Attack Tools:**
    1.  **Auth Flood:** Floods target Access Points with authentication frames to exhaust resources and force disconnections.
    2.  **CSA Disconnect (Channel Switch Announcement):** Transmits modified beacon frames informing target clients that the AP is switching channels, instantly forcing them to disconnect and look for a new AP.
    3.  **Deauth Flood:** Blasts target networks and client MACs with standard deauthentication packets.
    4.  **Beacon Flood:** Cycles channels 1–11 to transmit fake SSIDs, cluttering the local Wi-Fi scan list.
    5.  **Probe Request Flood:** Floods the airwaves with search requests to evaluate the responsiveness and detection capabilities of local intrusion detection systems (IDS).
*   **Integrated Evil Twin Scenario:**
    *   Clones a target SSID as an open AP on the target channel.
    *   Automatically runs an active CSA Disconnect attack against the real AP to push clients onto the clone.
    *   An embedded DNS Server intercepts and redirects all HTTP requests to a local Captive Portal (`router.update`).
    *   Admin receives buzzer siren feedback and instant attack completion upon capturing the credential.
*   **Physical RFID Authentication Lock:**
    *   **Secure State:** Hardware-arm/-disarm control via MFRC522 RFID keycard.
    *   **Tap-to-Unlock / Lock:** Prevents unauthorized web access by blocking core endpoints until a valid RFID card is tapped.
    *   **Double-Tap Emergency Stop:** Instantly aborts any active attack, shuts down clone APs, restores the management Wi-Fi, and tunes the antenna to a safe channel.
    *   **Intruder Lockout:** Tapping an unauthorized card sounds a low error beep and immediately locks down the dashboard.
*   **Management Auto-Lock:** Automatically locks the device API when the management phone or laptop disconnects from the AP, ensuring physical security.
*   **Modern Web UI Dashboard:** Built-in web portal containing network scanning, real-time metrics (packets sent, credentials captured), target selection, and custom attack configurations.

## ⚠️ Known Technical Bugs & Limitations

While the firmware implements several Wi-Fi frame injection routines, physical limitations of the ESP32 hardware and modern operating system security features prevent certain vectors from working reliably in the real world:

### 1. 2.4 GHz Antenna Constraint
*   The ESP32 radio chip is **2.4 GHz only**. It is physically impossible for the board to scan, monitor, or inject packets into the 5 GHz band. Networks broadcasting on 5 GHz cannot be targeted or disrupted by this hardware.

### 2. Protected Management Frames (PMF / 802.11w)
*   **Legacy Deauth / Disassociation Floods:** Modern access points and clients supporting **WPA3** or configured with **PMF (Protected Management Frames)** encrypt and cryptographically sign management packets (including deauthentication and disassociation frames). Consequently, clients and routers will silently drop unsigned deauthentication frames generated by the ESP32, rendering standard deauth floods ineffective on modern protected networks.
*   **Auth Floods:** Ram and buffer exhaustion attacks via authentication floods are blocked by modern APs that implement rate limiting on client association requests and track state sequences properly.

### 3. Client-Side Defensive Ignoring (CSA / Probe Attacks)
*   **CSA Disconnect:** While CSA (Channel Switch Announcement) is a powerful mechanism, modern operating systems (such as Windows 10/11, macOS, and recent iOS/Android updates) actively validate the validity of channel switches. If the client does not observe the corresponding channel-switch event or senses inconsistent beacon signals, it will ignore the CSA frame.
*   **Probe Floods:** Operating systems utilize MAC randomization for probe requests. Spammed probe requests from random MAC addresses are widely filtered out by modern intrusion detection systems (IDS) and are ignored by target APs.

## 💻 Software Installation

### 1. Library Dependencies
Open the **Library Manager** in your Arduino IDE or PlatformIO and install:
*   `MFRC522` (by Miguel Balboa)
*   Standard ESP32 Core libraries (`WiFi`, `WebServer`, `DNSServer`, `SPI`)

### 2. Configuration & Flashing
1.  Open the [esp32_Wifi_deauther.ino](file:///D:/Siddhesh%20Patil/Documents/Projects/esp32%20Wifi%20deauther/esp32_Wifi_deauther/esp32_Wifi_deauther.ino) sketch.
2.  Locate the authorized RFID UID array:
    ```cpp
    byte authorizedUID[] = {0x8B, 0x8E, 0x4E, 0x43};
    ```
    *If you don't know your card's UID, compile and upload the sketch first. Tap your card against the reader while connected to the Serial Monitor (115200 baud) to view your card's UID printed in hexadecimal, then update the array.*
3.  Choose your preferred management credentials:
    ```cpp
    const char* AP_SSID = "Management AP";
    const char* AP_PASS = "admin123";
    ```
4.  Compile and upload the firmware to your ESP32 board.

## 📖 Operational Guide

### 1. Powering On
*   Upon boot, the system initializes the SPI bus, MFRC522 module, and the buzzer.
*   A single short startup beep is played, and the secure management Access Point starts.
*   By default, the device is **Locked (Disarmed)**. Accessing the web API will return `403 Not Authorized`.

### 2. Unlocking and Arming (RFID Card Tap)
*   **Unlock:** Tap your authorized card against the MFRC522 reader. You will hear two high-pitched verification tones (`beep(1200, 150, 2)`), and the Serial Monitor will output `✓ Unlocked`.
*   **Lock:** Tap the authorized card again. You will hear a single medium tone (`beep(800, 400, 1)`), and the API is instantly locked.
*   **Intrusion Alert:** If an unregistered card is scanned, the buzzer will emit a low error tone (`beep(400, 600, 1)`) and immediately lock the device if it was previously armed.

### 3. Accessing the Dashboard
1.  Connect your phone, tablet, or laptop to the SSID **Management AP** (Password: `admin123`).
2.  Navigate to `http://192.168.4.1/` (or any address, as DNS redirection is active).
3.  If unlocked, the **ESP32 Security Dashboard** will load. Here, you can trigger an instant site scan, select targets, choose attack profiles, and start audits.

### 4. Running an Evil Twin Captive Portal
1.  Scan the networks, select a target Wi-Fi SSID, and arm the suite using your RFID card.
2.  Start the **Evil Twin** attack.
3.  The ESP32 changes its SSID and channel to clone the victim AP, while launching a parallel **CSA Disconnect** sequence to deauthenticate clients from the real network.
4.  Clients trying to reconnect will see the open cloned AP and connect. Upon attempting to load any HTTP web page, they are redirected to a router firmware update page asking for the WPA2 password.
5.  Once the victim submits a password, the ESP32:
    *   Prints the captured password to the Serial console.
    *   Saves the password to the dashboard status page.
    *   Sounds a distinct siren alert pattern (`beep(3000, 200, 5)`).
    *   Deactivates the Evil Twin AP and restores the **Management AP** so you can safely connect back and inspect the log.

### 5. Emergency Stop
If you need to instantly stop any ongoing attack or Evil Twin broadcast:
*   **Double-tap** or simply hold your authorized RFID card to the reader.
*   The buzzer will sound a double-tone, and the ESP32 will terminate the injection loops, spin down the cloned open AP, re-host the secure **Management AP**, and switch the radio to its home channel.

## 🛠️ Code Structure

*   **`setup()`**: Configures SPI, pins, custom transmit power, starts the captive DNS server, and boots the Web Server.
*   **`loop()`**: Continuously runs the captive portal DNS resolution, web server client handler, RFID polling, and the non-blocking attack engine ticks.
*   **`checkRFID()`**: Handles hardware card presence checks, validates the UID byte array, and triggers arm/disarm/emergency stop actions.
*   **`handleAttack()`**: The non-blocking execution engine. Uses FreeRTOS slice times to partition CPU time between high-rate packet injection (deauth, authentication, and probe floods) and HTTP page serving.
